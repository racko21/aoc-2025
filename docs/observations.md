### 2026-04-06 — Day 2: Gift Shop

Day 2 required finding "repeated-digit" numbers within given ranges and summing them. An invalid ID is a number whose digit string is some sequence of length k repeated m times (m≥2 for Part 2, m=2 for Part 1). The key mathematical insight: such a number equals `n * (1 + 10^k + 10^(2k) + … + 10^((m-1)k))` for a k-digit prefix `n`. Rather than scanning every integer in each range (which could be enormous), the solution generates all candidate repeated numbers up to the max range value, then checks membership. This kept the solution fast regardless of range size. Part 2 required extending the inner loop over repetition counts and deduplicating via a set (since e.g. `111111` is both "11" repeated 3 times and "111" repeated 2 times). `MustParseInt64` was added to utils to handle large 64-bit inputs. ### 2026-06-04 — Day 10: Factory

Day 10 split into two structurally distinct optimisation problems sharing the same input format. Part 1 (answer: 411) was minimum-weight solution over GF(2) — toggle buttons to match a target light pattern. This is equivalent to minimum-weight codeword finding, solved here with BFS over 2^N states (feasible because the number of lights per machine is small). Part 2 (answer: 16048) changed the semantics completely: buttons now increment integer counters, and the goal is to reach specified joltage targets with fewest total presses. This is a non-negative integer LP: min Σx_j subject to Ax=b, x≥0.

The most significant aspect of this day was implementing a Big-M simplex solver from scratch in Go (~80 lines). The problem required recognising the LP structure, setting up the Big-M objective row with the correct sign convention (the RHS of row 0 stores −z, not +z — a subtle point that causes silent wrong answers if inverted), and using Bland's rule for tie-breaking to guarantee termination without cycling. The LP relaxation gave exact integer solutions for all machines in the real input, confirming the AoC instances have integer LP vertices despite the general problem being NP-hard. No branch-and-bound was needed.

Notable for thesis: this is the first day requiring a non-trivial mathematical algorithm (LP/ILP) that is typically associated with operations research rather than competitive programming. Part 2 required four attempts and extensive debugging of the numerical LP implementation. Three distinct bugs were found sequentially: (1) incorrect rounding of fractional LP values (math.Round instead of B&B); (2) upper-bound constraints added as equalities instead of inequalities with slack variables, causing wrong sub-problems; (3) degenerate Phase-1 artificials corrupting Phase 2 by remaining in the basis and causing the simplex to produce "integer" solutions that violated the equality constraints. The final solution drives out degenerate artificials before Phase 2 and verifies all candidate integer solutions against Ay=b. This day is the most algorithmically complex of the set, requiring deep knowledge of LP theory and numerical methods. Claude Code correctly diagnosed each bug but required user feedback (wrong answer from the AoC grader) to iterate — it could not self-detect that its solutions were wrong without external oracle feedback. The final code is ~320 lines, significantly longer than typical AoC solutions.

### 2026-06-13 — Day 12: Christmas Tree Farm

Day 12 was a polyomino packing problem: given a set of shapes (each within a 3×3 bounding box, with '.' cells transparent) and a list of W×H regions, determine how many regions can fit all required present copies without '#'-cell overlaps. Shapes can be rotated and flipped (up to 8 orientations each).

The naive approach — backtracking over all placements — was implemented first and passed the example in 0.73s for 3 regions. It would be far too slow for 1000 real-input regions of up to 50×50 with 200+ presents each. The key insight came from input analysis: (1) every infeasible region is caught by the simple cell count check (total '#' cells > W×H); (2) every region that passes the cell count check also satisfies presents ≤ ⌊W/3⌋×⌊H/3⌋ non-overlapping aligned 3×3 slots — meaning all such regions are trivially packable by assigning each present to a unique aligned block. The final algorithm runs in ~73ms for all 1000 regions.

Notable for thesis: this is the clearest example so far of mathematical input analysis revealing that a computationally hard general problem (bin packing / exact cover) reduces to a trivial condition for this specific dataset. Claude Code initially went deep into algorithmic complexity (DLX, profile DP, coloring arguments) before stepping back and testing the "aligned slots" hypothesis empirically. The backtracking fallback was kept for correctness on the example (which includes a region where the cell count passes but packing is genuinely infeasible — requiring actual search). The final design uses a layered approach: fast O(1) checks first, expensive search only as fallback. Part 2 was not yet available.

### 2026-06-23 — Day 7 (C): Laboratories

Part 1 asks how many splitters are actually hit, which maps naturally onto a row-by-row simulation over a boolean "active columns" set — for each row, any active column sitting on a `^` increments the hit counter and spawns left/right columns in the next row's set, while columns on `.` simply carry straight down; set semantics (OR, not count) give the natural beam-merging behavior for free. This is the same row-by-row simulation but with the boolean set replaced by per-column `long long` multiplicities that *add* on merge (rather than collapsing to one), and the final answer is just the sum of all column counts after the last row — since timelines only ever double (split) or pass through unchanged, that sum is invariant once the last splitter row is processed. Recognizing that Part 2 was the identical traversal with substitution of OR-merge for ADD-merge made it a very fast extension once Part 1's structure was in place — no algorithmic redesign needed, just changing the per-column value type from `bool` to `long long`. ### 2026-06-23 — Day 9 (C) continuation: Part 2

A follow-up session received Part 2's text once it was available. Part 2 restricts the rectangle to only red+green tiles, where green tiles are the straight connecting segments between consecutive red tiles in input order (the sequence forms a closed rectilinear polygon boundary) plus everything enclosed by that loop. Since real coordinates run up to ~98000, a literal grid is infeasible, so the solution uses coordinate compression: each of the ~500 unique vertex x/y values gets its own singleton row/column in a compressed grid, and the constant-status gap between two consecutive vertex values collapses into one weighted row/column (storing its true width). Polygon edges are then rasterized onto this small (~991×991) compressed grid as boundary cells, a BFS flood fill from a padded exterior corner classifies every other cell as outside-the-loop, and a 2D prefix sum over true cell area (row weight × column weight) answers "is this candidate rectangle's bounding box fully red/green?" in O(1) per pair, by comparing the summed allowed-area inside the box against its total real area. All ~123k pairs are checked this way; the candidate maximizing total area while passing this containment test wins. Verified against the example (24, between `9,5` and `2,3`) before running on the real input (1539809693). ### 2026-06-23 — Day 10 (C) continuation: Part 2

A follow-up session received Part 2's text directly from the user (the same `PROBLEM.md` gap as Part 1's earlier "no Part 2 exists" conclusion turned out to be premature — the puzzle does have a Part 2, just not auto-captured into the file). The lever flips button semantics from XOR-toggle to additive: each press now increments every counter a button lists by 1, and the goal is to reach an exact joltage vector (parsed from the previously-ignored `{...}` values) with the fewest total presses. This is `min sum x_j s.t. A x = b, x_j >= 0 integer`, where `A` is the same 0/1 button-to-counter incidence matrix used for Part 1's XOR masks, just reinterpreted arithmetically. The worked example is the key structural clue: machine 1's optimal solution presses 5 distinct buttons to satisfy a 4-counter, 6-button system — more nonzero variables than the system's rank (≤4) — which rules out the tempting shortcut of enumerating only rank-sized basic feasible solutions (i.e. LP vertices); the true integer optimum can require a strictly larger support than any LP vertex achieves. This forced implementing a genuine integer program: branch-and-bound over a Big-M simplex LP relaxation. Each equality row (one per counter) gets an artificial variable penalized at a large cost `M = 1e7` (safely above the largest observed per-machine press total, ~1754, by several orders of magnitude); the branch-and-bound layer tightens per-button `[lo, hi]` press-count bounds on the most fractional variable at each LP-optimal node, folding lower bounds into the right-hand side via a `y_j = x_j - lo_j` shift (so no extra row is needed for them) and adding a slack row only for newly-imposed upper bounds. Pruning compares each node's LP relaxation cost (a valid lower bound, since integrality constraints can only raise the true optimum) against the best integer solution found so far. One C-specific snag: the initial implementation called `floor()`, but the shared `c/Makefile` doesn't link `-lm`, causing an undefined-reference link error; rather than touch the cross-day Makefile, a `floor_nonneg()` helper using a direct `(long long)` cast suffices here since every press count handled is non-negative and well within range. Verified against the example (33 = 10+12+11) before running on the real input (16063). 0 compiler warnings, 441 LOC total for `main.c`, 1.02ms runtime and 1.8MB peak RSS for Part 2 at `-O2` — the branch-and-bound tree stays small at this problem size (≤10 counters, ≤13 buttons per machine), so no heap allocation was needed anywhere in the ILP solver; the simplex tableau is a fixed-size stack array.

### 2026-06-23 — Day 11 (C): Reactor

As with most prior C days, `PROBLEM.md` only contained Part 1's text at session start; Part 1 was solved and verified first, then the user pasted Part 2's statement manually mid-session. The problem is a directed-acyclic device graph (each line lists a device's outputs; "data only ever flows forward") and Part 1 asks for the number of distinct paths from `you` to `out`. With ~576 nodes and a real possibility of exponential path counts, the natural approach is a memoized DFS: the number of paths from any node to `out` depends only on that node (since the graph is acyclic, no cycle can re-enter a node's subtree), so `count_paths(node)` recurses over outgoing edges and caches per-node results in a `long long` array. Part 2 changes the start node to `svr` and adds a constraint: only count paths that visit both `dac` and `fft`. This generalizes cleanly by widening the memoization key from `node` to `(node, has_dac, has_fft)` — the two boolean flags are updated immediately upon entering a node (so a node satisfies its own flag before checking the `out` base case), and since the DAG plus two extra boolean bits is still acyclic and bounded, memoizing per `(node, flags)` combination remains valid and keeps runtime linear in `nodes * 4` rather than exponential in path count. One real bug surfaced during Part 1 development: the parser used two independently-nested `strtok()` calls (an outer loop splitting lines on `\n`, an inner loop splitting each line's destinations on `' '`), which silently share a single static internal cursor across both call sites — the inner loop's calls clobbered the outer loop's resumption point, causing the line iteration to terminate after the first line and producing a wrong answer of 0 paths with no compiler warning or crash to flag it. This is a classic C footgun invisible in Go (where `strings.Split`/`bufio.Scanner` have no shared global state) and was diagnosed by adding an unbuffered `fprintf(stderr, ...)` print before the failing `assert` — the first `printf` attempt printed nothing at all because `assert()`'s `abort()` does not flush stdio buffers, a second, more subtle reminder that C's buffered I/O can silently swallow diagnostic output exactly when a crash makes it most needed. Fixed by switching both loops to `strtok_r()` with independent save-pointer variables, which in turn required `#define _POSIX_C_SOURCE 200809L` before the includes since plain `-std=c17` doesn't expose the POSIX reentrant variant. Both parts verified against their respective examples (5 paths; 2 paths visiting both `dac` and `fft`) before running on the real input (523; 517315308154944 — the large Part 2 count reflecting how many of the exponentially-many `svr`-to-`out` paths happen to pass through both named nodes). 0 compiler warnings, 177 LOC, ~1.5ms runtime and 1.8MB peak RSS for both parts at `-O2`. No `malloc`/`free` imbalance: the file buffer and both memo arrays are freed at the end of each solve function; `reset_graph()` frees all adjacency-list allocations before each part re-parses the input (the graph is parsed twice, once per part, since `svr` requires Part 2's input augmented with no Part-1-only assumptions).

### 2026-06-23 — Day 12 (C): Christmas Tree Farm

`PROBLEM.md` for this, the final calendar day, only describes a single task (count how many regions can fit all required shape quantities) — the user confirmed there is no Part 2 for day 12, so this was a one-part day. The puzzle gives 6 polyomino shapes on 3x3 grids (`#` = blocking, `.` = transparent/non-blocking) and 1000 `WxH` regions, each demanding a quantity of every shape; shapes may be freely rotated and flipped, and only `#`-vs-`#` overlap between placed pieces is forbidden — `.` cells of one piece may freely sit under another piece's `#`. Each shape's distinct orientations (rotation x flip, deduplicated by symmetry — anywhere from 2 to 8 unique per shape depending on how symmetric the 3x3 pattern is) are precomputed once at parse time. For each region, the required pieces are flattened into a multiset ordered largest-cell-count-first and placed via recursive backtracking that tries every orientation and position, backtracking only on `#` collision, with two prunes added to keep the search tractable: a free-area-vs-remaining-required-cells cutoff, and a global node-visit budget (2,000,000) as a safety valve against combinatorial blowup when a region is genuinely infeasible (proving infeasibility requires exhausting the search tree, which is expensive at this scale — up to ~200 pieces on a 50x47 grid). A temporary instrumented run confirmed the budget was never actually exhausted on the real input, so all 481 positive results are exact rather than a heuristic guess. The first parser bug came from using `sscanf(line, "%d:", &idx)` to detect shape-definition headers (`"4:"`); this also silently matches region lines like `"12x5: 1 0 1 0 2 2"`, because `sscanf` returns the count of successful conversions *before* a literal mismatch — it had already converted "12" before failing to match the literal `:` against `x`, so it returned 1 and the region line was misparsed as an empty shape definition, eventually consuming the entire region list and producing Part 1 = 0. The fix requires the entire line to match `^\d+:$` exactly rather than just a leading-digit prefix. A second, more interesting detour: a debug (`-O0`) build of the full-backtracking solver appeared to hang indefinitely on the real input, which looked like an algorithmic dead end and prompted a (wrong) pivot to a non-backtracking greedy first-fit placer; greedy was fast but incorrect even on the tiny 4x4 worked example (it declared 2 copies of one shape infeasible in a case that does have a valid interlocking placement, since greedy commits to the first position found for the first piece without ever reconsidering it). Reverting to budgeted backtracking with area pruning resolved both problems at once: it runs in 67ms on the real (release, `-O2`) build, meaning the earlier "hang" was simply slow unbounded backtracking in an unoptimized debug binary on one difficult region, not a fundamentally intractable search. This is a useful C-specific contrast with the Go arm: in Go, a similarly naive backtracking solution would likely have been tried directly in a release-equivalent build from the start (`go run` has no separate debug/release distinction to obscure a slow path), whereas here the `-O0`/`-O2` split briefly created a false signal of infeasibility. 0 compiler warnings, 298 LOC, 67.29ms runtime and 1.86MB peak RSS at `-O2`. No `malloc`/`free` imbalance: only the line-split array and each region's `occupied` grid are allocated and freed.
